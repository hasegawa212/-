## 1. 業務概要

### 業務名
銀行事前申込（事前承認）対応

### 目的
売買契約前に、買主のローン審査が通る見込みを銀行と保証会社に確認する手続き。事前承認が出ないと売買契約には進めない。

### 頻度・タイミング
- 各案件で 1 回（売買契約前）
- 標準リードタイム：書類提出 → 事前承認まで 1〜2 週間

### 必要なツール・アクセス権限
- 銀行担当者の連絡先（CRM 銀行マスタ参照）
- 申込書類一式（テンプレート: 各銀行で異なる、銀行マスタに保管）
- 顧客の本人確認書類・収入証明
- 物件資料一式

---

## 2. 標準フロー（14 ステップ）

### ステップ 1：挨拶・書類提出
- 銀行担当者へ挨拶
- 事前申込書類一式を提出
- **書類不備チェック**：
  - 本人確認書類の有効期限（運転免許証など、失効がないか必ず確認）
  - 申込書の記入漏れ（生年月日・住所・勤務先など）→ その場で確認・追記

### ステップ 2：物件の現況説明
- 物件の現状（リフォーム前 / リフォーム済 / 既存居住者の有無）を正確に説明
- リフォーム前提案件の場合、**「販売は完成形（リフォーム後）の状態」**であることを明示
- 価格に何が含まれるか（物件 + リフォーム + 保証 + 保険等）を一括説明

### ステップ 3：居住状況・契約状況の確認
- 既存居住者の有無、退去予定時期
- 契約フローを時系列で説明：
  ```
  事前承認 → 売買契約 → 既存居住者退去 → 決済
  → リフォーム → 引渡し・入居
  ```
- ※ 本申込・金消契約のステップは案件によって挿入位置が異なる

### ステップ 4：購入動機の説明（保証会社審査用）
- 保証会社の審査では購入動機が重要
- 用意すべき情報：
  - 同居予定者・家族構成・将来計画
  - 完済目標年齢（多くの場合 60 歳前後）
  - 返済期間とライフプランの整合性

### ステップ 5：物件の担保評価情報
- 銀行が確認するポイント：
  - 築年数
  - 接道状況
  - 建物状態（外観・室内）
  - リフォームが必要な理由（ペット痕跡・水回り老朽化等）
- 担保評価が下振れしそうな要素は**先回りして説明**（後出しすると印象悪化）

### ステップ 6：申込人の属性・他行審査状況
- 銀行が確認するポイント：
  - 勤務先（民間 / 公務 / 法人区分）
  - 職業・職位
  - 会社規模
- 他銀行への審査状況は**正直に開示**（隠すと信用情報照会で発覚）

### ステップ 7：決済時期・登記の流れ
- 決済希望時期（月単位で）
- 同日決済の標準フロー：
  - 売主への支払い
  - 買主の融資実行
  - 所有権移転登記
  - 司法書士が両当事者の書類を預かり同時処理
- 決済後のスケジュール：
  - 鍵引渡し → リフォーム工事（標準 1 ヶ月以内）→ 入居
- 例：6 月末決済の場合、7 月末前入居

### ステップ 8：販売形態の合理性説明（リフォーム込み販売の場合）
- 銀行から「なぜリフォーム込み販売か」と質問される定番ケース
- 説明テンプレート：
  - 顧客が別途リフォームローンを組む負担を減らせる
  - 業者保証を付けられる
  - 工事費の細かいやり取りを顧客側で行う必要がない
  - 完成形での価格明示で安心感
  - 一本化された資金計画

### ステップ 9：資金計画の説明
- リフォーム込み総額での月々返済額を顧客に確認済であることを伝える
- 金利前提：
  - 変動金利前提（多くの案件）
  - 金利目安：直近のマーケットレンジ（2026 年時点で 1% 台前半 など、随時更新）
  - 多少の前後は許容ラインとして共有
- ※ 金利水準は**マーケット変動に応じて最新値を採用**（マニュアル四半期更新項目）

### ステップ 10：物件仕入れルートの説明
- 銀行から「こういった物件をどう入手しているか」と質問される定番ケース
- 説明テンプレート：
  - 流通機構（レインズ等）掲載物件
  - 取引先業者からの紹介
  - 表に出る前の水面下情報
  - 現地確認・過去取引先からの情報
  - 顧客希望条件とのマッチング
- 自社の立ち位置：**単純仲介ではなく「プランニング型」**（物件探索 + リフォーム + 保証）
- → 銀行に「単発仲介ではなく事業者として継続性がある」印象を与える

### ステップ 11：銀行選定理由・過去取引の話
- なぜこの銀行に相談しているかを言語化：
  - エリア対応性（物件所在地が銀行の融資エリア内）
  - 過去取引の関係性
- **「むやみに複数行に審査を出していない」**ことを明示（信用情報の汚染を避ける配慮）
- 過去案件の話で関係性確認・雑談（信頼構築）

### ステップ 12：団信・金利条件の確認
- 標準：一般団体信用生命保険
- 特約（3 大疾病等）は金利上乗せのため、案件の方針に応じて判断
- 上乗せなしで使える保障内容は銀行ごとに異なる → 確認しておく

### ステップ 13：本人確認電話・連絡体制
- 保証会社・銀行から申込人本人へ確認電話が入る可能性
- 申込人が知らない番号を警戒するタイプの場合、**事前にこちらから案内**
- 銀行からの確認事項は**営業担当（嶺田）宛に集約**する流れを作る

### ステップ 14：雑談・関係構築
- 過去販売物件、表札、居住確認、現地状況などの雑談
- **「ただの書類提出」ではなく「関係構築の場」**として活用
- 担当者との信頼関係は次案件のスピードに直結

## 3. 銀行が見ているポイント（要約）

銀行担当者・保証会社が確認したい本質：

| 観点 | 確認内容 |
| --- | --- |
| 案件の仕組み | この物件はどういう仕組みで販売されているか |
| 価格の妥当性 | なぜこの金額なのか（リフォーム + 保証含む内訳） |
| 関係者の役割 | 誰がどのタイミングで動くか |
| 責任範囲 | リフォーム・保証は誰の責任で行うか |
| 申込人の返済能力 | 属性・収入・他行審査・ライフプラン |
| 担保評価 | 物件の現況・築年数・接道・建物状態 |
| 売主／買主／業者の関係性 | 利益相反がないか |

→ **これらを先回りして説明できると審査スピードが上がる**

## 4. 失敗パターン（過去の学び）

| ❌ パターン | 🟢 是正策 |
| --- | --- |
| 書類不備（記入漏れ・期限切れ）に銀行で気づく | 提出前に自社でダブルチェック |
| リフォーム前提を後出し | 最初の説明から「完成形販売」と明示 |
| 担保下振れ要素を隠す | 先回り説明で信頼確保 |
| 他行審査状況を隠す | 正直開示（信用情報で必ず発覚） |
| 複数行同時申込 | エリア・過去取引で 1 行に絞る |
| 申込人へ事前案内なしで本人確認電話 | 事前案内 + 営業担当者経由 |
| 雑談を軽視 | 関係構築の場として活用 |

## 5. 銀行マスタ（CRM 連携項目）

各銀行で以下を CRM に蓄積（脱属人化原則 ③ — 顧客は会社が保有）：

- 銀行名・支店名
- 主要担当者（複数）
- 過去取引案件
- 融資エリア
- 申込書テンプレート（最新版）
- 審査スピード（標準・最速・最遅）
- 金利水準・特約条件
- 保証会社の連絡先
- 担当者ごとの好み・コミュニケーション傾向
