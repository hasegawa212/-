# 案件 ID 採番 + 案件管理台帳 運用ルール v1.0（暫定）

> 作成日: 2026-05-16
> 主担当: **三浦 一通**（仕入れ会議議長として運用責任）
> バックアップ担当: **藪**（記録・台帳メンテ実務）
> 派生元: [仕入れ会議運営マニュアル §4](../仕入れ・案件管理/仕入れ会議運営マニュアル.md)
> 脱属人化スキーム該当原則: ②知識の文書化、③顧客は会社が保有

---

## 1. このルールの目的

社内・社外のあらゆる場所で**案件を一意に特定**できる ID 体系を確立する。

これにより：
- 議事録・チャット・書類で**顧客実名を出さずに**案件を指せる（PII 漏洩リスク低減）
- 過去案件の参照が容易になる
- マニュアル間の相互参照が可能になる

---

## 2. 案件 ID フォーマット

```
YYYY-MM-NNN[-X]

YYYY  : 案件発生年（4 桁）
MM    : 案件発生月（2 桁）
NNN   : 月内通番（001 から、3 桁ゼロ埋め）
X     : 区分サフィックス（オプション、下記参照）
```

### 例
- `2026-05-001` — 2026 年 5 月の 1 号案件
- `2026-05-001-B` — 同一案件で別物件提案に派生したもの
- `2026-05-001-T` — 同一案件で系列の別取引（紹介・追加案件等）

### サフィックス区分

| サフィックス | 意味 | 使用ケース |
| --- | --- | --- |
| なし | メイン案件 | 標準 |
| -B | 派生物件提案 | 同一顧客で物件を変更 |
| -T | 関連取引 | 同一顧客の追加取引・紹介派生 |
| -R | リプロセス | トラブル後の再開案件 |
| -X | 失注・破談 | 履歴保管のため案件 ID は残す |

---

## 3. 採番タイミング

**案件 ID は「最初の社内記録」が発生した瞬間に採番**：

| トリガー | 採番者 |
| --- | --- |
| 初回問合せ受領 | 受領者（受付・営業誰でも） |
| 紹介案件の初回情報入手 | 営業担当 |
| 仕入れ会議で初出 | 書記（藪）が即時採番 |
| 業者からの物件持込 | 業者対応者 |

→ **「とりあえずメモ」レベルでも案件 ID を振る**。後から振り直す手間を防ぐ。

### 採番の重複防止

- 採番は**案件管理台帳に行を追加した瞬間**に確定
- 台帳の「最新採番」セルから自動採番（スプレッドシートなら関数）
- 同時採番が発生した場合は**より早く台帳更新した方を有効**とする

---

## 4. 案件管理台帳（マスタ）の構造

案件管理台帳は**全案件の中央レジストリ**。最低限 15 列：

| # | 列名 | 必須 | 例 |
| --- | --- | --- | --- |
| 1 | 案件 ID | 必須 | 2026-05-001 |
| 2 | 案件発生日 | 必須 | 2026-05-15 |
| 3 | 顧客ハンドル | 必須 | 顧客 A（実名は別管理） |
| 4 | 顧客実名 | 機微 | （別シート / アクセス制限） |
| 5 | 物件概要 | 必須 | エリア・種別のみ（住所は別管理） |
| 6 | 営業担当 | 必須 | 社員名 |
| 7 | ローン付け担当 | 必須 | 嶺田 / 他 |
| 8 | 利用銀行 | 任意 | 銀行マスタ参照 ID |
| 9 | ステータス | 必須 | 標準語彙 11 種（仕入れ会議マニュアル §5） |
| 10 | 想定決済月 | 必須 | 2026-06 |
| 11 | 想定粗利 | 任意 | 円 |
| 12 | 直近アクション | 必須 | 何を、いつ |
| 13 | 次タスク・期限 | 必須 | 誰が、何を、いつまで |
| 14 | リスク・懸念 | 任意 | 自由記述 |
| 15 | 関連 ID | 任意 | -B / -T / -R / -X の親 ID |

---

## 5. 「ハンドル名」運用ルール

PII 漏洩を防ぐため、社内チャット・議事録では**ハンドル名で運用**：

### 命名ルール
- `顧客 A`、`顧客 B`、`顧客 C` … （シンプルな英字記号）
- または **案件 ID そのもの**を使う：「2026-05-001 の件」

### 禁止
- ❌ 実名（フルネーム）
- ❌ 名字 + 業界知名度
- ❌ 役職 + 会社名（特定可能）

### 例外
- 銀行担当者と話す場面で実名は使用可（業務上必要）
- 公式書面・契約書は当然実名
- → **チャット・議事録 = ハンドル / 書面 = 実名**の使い分け

## 6. 採番と参照の流れ

```
[1] 案件発生
   ↓ 採番（その場）
[2] 案件管理台帳に行追加（ID / 日付 / ハンドル / 物件概要 / 営業 / ステータス：見込み）
   ↓
[3] 社内チャット・議事録：「案件 2026-05-001 について」
   ↓
[4] 銀行・司法書士等 外部書面：実名利用（台帳の「顧客実名」シート参照）
   ↓
[5] 仕入れ会議で進捗報告（案件 ID 順）
   ↓
[6] ステータス変化のたびに台帳更新（責任：営業担当）
   ↓
[7] 決済完了 or 失注 → ステータス確定、関連ファイルアーカイブ
```

---

## 7. 台帳の保管場所

⚠️ **顧客実名・物件住所・財務情報を含むため、git にコミットしない。**

推奨：
- **Google Workspace 限定共有スプレッドシート** または **Notion データベース**
- 編集権限：営業 + ローン付け + 書記
- 閲覧権限：経営層 + 上記
- **「顧客実名」列は別シート分離 + アクセス制限**

---

## 8. 過去案件の参照

- アーカイブ後も**案件 ID は永続**（再利用不可）
- 同一顧客の追加取引は新 ID を採番（既存 ID にぶら下げない）
- 過去取引履歴の検索：台帳の「顧客実名」シート or 銀行マスタの「過去取引履歴」

---

## 9. 採番ミス・修正ルール

| ミス | 対応 |
| --- | --- |
| 採番後にすぐ気付いた | 行削除 → 再採番（NNN は欠番のままでも可） |
| 既に通信に使った後で気付いた | **元 ID 維持**、誤記録は「note 列」に注記 |
| 重複採番（複数人が同時採番） | 早い方を有効、遅い方は次番号に繰下げ |
| 案件取消し | ステータス「失注」+ -X サフィックス追加、ID は残す |

→ **欠番・再利用は厳禁**。履歴の整合性を保つ。
