## 1. このガイドの位置づけ

進行中の住宅ローン審査案件を **5 ステージのパイプライン** で可視化・追跡するための運用ルール。

**実データ自体は git にコミットしない**（顧客名・物件・金融機関の組合せは PII）。
Google Sheets 上で運用し、本ガイドは **フォーマット・更新ルール・分類基準** だけを明文化する。

既存マニュアルとの関係:

- 案件 ID 採番・全体台帳ルール → [案件 ID 採番・管理台帳ルール](./案件ID採番・管理台帳ルール.md)
- 各マスタ全体構成 → [情報資産マスタ 運用ガイド](./情報資産マスタ運用ガイド.md)
- 工程別必要書類 → [住宅ローン業務 全工程チェックリスト](../決済対応/住宅ローン業務全工程チェックリスト.md)

## 2. パイプライン 5 ステージ

```
[商談中・銀行検討中]
    ↓
[事前審査・申込済]
    ↓
[本承認・本審査中]
    ↓
[金消契約完了]
    ↓
[案件完了（決済 + 引渡し）]
```

### 2.1 各ステージの定義（脱属人化原則⑤）

| ステージ | 定義 | 出口条件 |
| --- | --- | --- |
| **商談中・銀行検討中** | 顧客との初回〜複数面談中。申込銀行未確定 | 申込銀行を 1 行以上確定 |
| **事前審査・申込済** | 1 行以上に事前申込を投入済み | 事前承認を 1 行以上取得 |
| **本承認・本審査中** | 事前承認後の本審査ステージ | 本承認取得 |
| **金消契約完了** | 金銭消費貸借契約締結済み | 決済日確定 |
| **案件完了** | 決済完了・物件引渡し完了 | （終了） |

### 2.2 ステージ滞留時間の目安

| ステージ | 標準所要日数 | アラート閾値 |
| --- | --- | --- |
| 商談中 → 事前申込 | 2 週間以内 | 1 ヶ月超 |
| 事前申込 → 本承認 | 1 〜 3 週間 | 4 週間超 |
| 本承認 → 金消 | 2 〜 4 週間 | 5 週間超 |
| 金消 → 決済 | 1 〜 4 週間 | 5 週間超 |

→ アラート閾値超過案件は **週次レビューで個別追跡**

## 3. Google Sheets フォーマット（必須列）

> 実シート例: 「銀行審査状況一覧【完全版】」(2026-05-16 版)

| 列 | 説明 | 例 |
| --- | --- | --- |
| **案件 ID** | 案件 ID 採番ルール準拠（`DEAL-YYYY-00001`） | DEAL-2026-00045 |
| **区分** | パイプラインステージ | 5 月決済 / 事前審査 / 6 月予定 / 商談中 |
| **顧客名** | 個別案件管理用 | （PII のため伏せ字 or 別シート分離） |
| **申込銀行** | 確定済み銀行 | 中央労働金庫 / 水戸信用金庫 |
| **担当** | 社内担当者 | 坂本 / 三浦 |
| **次のアクション** | 直近 1 週間で起こすべき動き | 本審査持込予定 4/22 |
| **アラート** | 閾値超過なら ✗ | — / ✗ |
| **最終更新日** | 自動 or 手動 | 2026-05-16 |

### 3.1 顧客名 PII 分離ルール

- メインシートには **案件 ID + 区分 + 申込銀行 + 担当 + アクション** のみ
- **顧客実名 + 物件詳細 + 連絡先** は別シート「顧客実名サブシート」に格納
- 顧客実名サブシートは **アクセス権限制限**（CEO + 主担当 + バックアップのみ）

→ 既存ルール [情報資産マスタ 運用ガイド §3](./情報資産マスタ運用ガイド.md) に準拠

## 4. 更新頻度・運用ルーチン

### 4.1 必須更新タイミング

- 申込銀行確定時
- 事前申込投入時
- 事前承認 / 否決取得時
- 本申込投入時
- 本承認 / 否決取得時
- 金消完了時
- 決済完了時

### 4.2 週次レビュー（CEO + 主担当）

- 毎週金曜 16:00 開催
- アラート ✗ 案件を全件レビュー
- ステージ滞留の原因究明・次手協議

### 4.3 月次サマリ報告（経営層 No.2 採用後は経営層 No.2 主導）

- ステージ別案件数
- 月次決済予定 / 実績
- 否決案件数 + パターン分析 → [住宅ローン シミュレーターの否決ケース DB](../../../ultimate-ai-agent/server/loanSimulator/rejectionCases.ts) 更新

## 6. 否決時の運用

詳細は [住宅ローン シミュレーター] の否決パターン学習結果を参照。

否決を受けたら：

1. **24 時間以内に否決理由を構造化記録**（タグ分類）
2. シミュレーターの推奨ルートに沿って **次の申込銀行を 1〜2 行に絞る**
3. **連続 3 件否決した場合は方向性再検討**（属性改善・物件種別変更）
4. 月次サマリで **否決傾向を経営層レビュー**

否決タグ分類:

- `property_road_access`: 接道義務違反
- `property_atemono`: 充て物審査
- `property_kakaku`: 物件価格不適合
- `borrower_employment`: 雇用契約不備
- `borrower_single`: 単身者懸念
- `borrower_dansin_kokuji`: 団信告知事項
- `borrower_kojin_shin`: 個信記録
- `borrower_all_failed`: 全行・全保証会社で否決
